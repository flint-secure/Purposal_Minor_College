\subsection{System Design Diagrams}

The following figures summarise the proposed architecture.

\newpage
\usecasefigure[0.95]{uml-use-case}{Use case diagram (Flint Secure system boundary).}
Figure~\ref{fig:uml-use-case} defines the planned actors and their goals. The \textbf{End User Device (SDK)} and \textbf{Merchant Developer} will call \emph{Identify devices}, which includes validating the API key, matching an existing device, and storing metadata; \emph{Recover device drift} extends identification when fingerprints change. The merchant developer will also \emph{Manage API Keys} (create, list, revoke). The \textbf{Fraud / Risk Analyst} will \emph{Review alerts}, optionally \emph{report fraud} or \emph{inspect identity risks}. The \textbf{PSP Backend} will \emph{Score Transaction}, including fetching device signals and calculating a risk score, then optionally publishing a scored event or flagging/blocking the payment.
\newpage

\umlfigure[1.0]{uml-seq-identify}{sequencediag}{Sequence diagram 1: device identification (actor: End User Device / SDK).}
Figure~\ref{fig:uml-seq-identify} shows the SDK identification path. The SDK sends \texttt{POST /v1/identify} to the API Gateway, which validates the API key through the Ingest service. Ingest looks up the fingerprint in Redis; on a cache miss it stores a new fingerprint record. Ingest enqueues an event to the Scoring service and immediately returns \texttt{202 Accepted} with an \texttt{eventId}. Scoring then resolves the fingerprint and computes a risk score asynchronously; Ingest updates the session score in Redis and notifies the SDK with \texttt{onBlock} or \texttt{onAllow} depending on the score threshold.
\newpage

\umlfigure[1.0]{uml-seq-keys}{sequencediag2}{Sequence diagram 2: API keys and webhooks (actor: Merchant Developer).}
Figure~\ref{fig:uml-seq-keys} covers merchant integration setup. In section~A, the developer creates an API key (\texttt{POST}, \texttt{201 Created}), lists keys (\texttt{GET /v1/apikeys}), and revokes a key (\texttt{DELETE}, \texttt{204 No Content}) via the API Gateway and Auth Service. In section~B, the developer registers a webhook endpoint (\texttt{POST /v1/webhooks}) through Ingest. In section~C, when a transaction is blocked, Ingest dispatches a \texttt{BLOCK\_TX} event to the Webhook service, which \texttt{POST}s the notification to the merchant; retries run if no acknowledgement arrives within five seconds, and failed deliveries are marked in Ingest after the maximum retry count.
\newpage

\umlfigure[1.0]{uml-seq-score}{sequencediag4}{Sequence diagram 3: transaction scoring and blocking (actor: PSP Backend).}
Figure~\ref{fig:uml-seq-score} describes the PSP scoring path. In section~A, the PSP Backend sends \texttt{POST /v1/score} with a transaction payload; the Gateway forwards it to Ingest, which fetches a feature vector from Scoring. Ingest evaluates rules to produce a combined score in parallel with background scoring work; if the score exceeds the threshold the PSP receives \texttt{BLOCK\_TRANSACTION}, otherwise \texttt{APPROVE\_TRANSACTION}. In section~B, Scoring dispatches an asynchronous webhook event (\texttt{txId}, score, flags) back to the PSP. In section~C, the PSP may confirm a block with \texttt{POST /v1/block/\{txId\}}, which updates the event status to \texttt{BLOCKED} and returns \texttt{200}.
