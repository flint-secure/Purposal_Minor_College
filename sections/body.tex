\section{Problem Statement}

In the last five years, mobile wallets such as eSewa, Khalti, and IME Pay have grown quickly in
Nepal. The Nepal Rastra Bank reports more than ten million mobile-banking users (2024). More
users means more fraud risk. Most payment service providers (PSPs) still block fraud with
simple rules (for example, a maximum amount or a daily limit) that cannot catch organised
attacks.

Three main gaps motivate this project:
\begin{enumerate}
  \item Rules are updated manually and do not adapt to new attacks such as SIM swap, stolen
        passwords, or fake devices.
  \item Transactions are not linked to a stable device, so one phone used for many accounts
        (money-mule behaviour) stays hidden.
  \item Systems rarely learn normal user habits (typical amounts, usual hours, usual recipients),
        so fraud is noticed only after money is lost.
\end{enumerate}

FIU-Nepal reports a large year-on-year rise in SIM swap fraud~\cite{fiu2024}. Nepal's FATF
grey-list status also pushes banks to improve monitoring. When controls are weak, fraud is
missed; when rules are too strict, honest customers are blocked and providers lose business.

\section{Project Objectives}

\subsection{Main Objective}
The \textbf{main objective} is to \textbf{design, implement, and test} Flint Secure: a
real-time fraud and device-intelligence platform for Nepali PSPs. The platform will provide
stable device identification across browser or app updates, risk scoring with SIM-swap
protection, and measurable speed targets (95\% of score requests under 50\,ms and about
3,200 requests per second in planned load tests). A large automated test suite (about 168
cases, at least 92\% pass) will be used to verify behaviour. Build steps and validation are
described in Sections~\ref{sec:methodology} and~\ref{sec:validation}.

\section{Significance of the Study}

International tools such as Stripe Radar, Sift, and Sardine are strong but costly for typical
Nepali wallet volumes and are not tuned for local problems like SIM swap. Flint Secure is
significant because it will offer \textbf{affordable, local} fraud infrastructure using open
software (Go, Redis, NATS, PostgreSQL) that a PSP can run on low-cost cloud hardware. The
project will also show that a minor-project team can meet clear performance and security
targets suitable for a pilot deployment.

\section{Literature Study/Review}
\label{sec:literature}

This section reviews prior work on device fingerprinting, payment fraud, velocity checks,
mobile account takeover, and the Nepali context. It supports the design in
Section~\ref{sec:methodology}. Together, these subsections form more than 15\% of the
proposal text, as expected for a proper literature review.

\subsection{Device Fingerprinting and Identity Stability}
Eckersley~\cite{eckersley2010} showed that browser signals can identify devices without
cookies. Laperdrix et al.~\cite{laperdrix2016} used canvas and graphics rendering to trace
hardware differences; this idea will guide our first-level fingerprint in the SDK.
Vastel et al.~\cite{vastel2018} studied how fingerprints change after updates; their work
supports our planned three-step server match: exact hash, recovery by stored device ID, then
fuzzy comparison when the fingerprint drifts. Separating device identification from transaction
scoring keeps the payment path fast.

\subsection{Statistical and Machine-Learning Fraud Detection}
Bolton and Hand~\cite{bolton2002} reviewed statistical fraud methods and ranked velocity
features (how often, how much, and to whom money moves) among the best real-time indicators.
We will store these counts in Redis for quick lookup on each score request. West and
Bhattacharya~\cite{west2016} surveyed machine-learning fraud systems; ensembles often work
better, but banks also need simple, explainable rules. Fiore et al.~\cite{fiore2019} discussed
handling rare fraud cases when training data is small. Dal Pozzolo et al.~\cite{dalpozzolo2015}
explained how to calibrate risk scores into clear bands. For this semester we will use
\textbf{rule-based scoring} with published weights; machine learning is not planned for
this semester's deliverable.

\subsection{Mobile Account Takeover and National Context}
Mjaaland et al.~\cite{mjaaland2021} described SIM swap attacks on mobile banking. Combined
with FIU-Nepal findings~\cite{fiu2024}, this justifies strong phone-to-device checks in our
scoring model. Better monitoring also supports national compliance goals, not only private
profit for one company.

\subsection{Commercial Platforms and Research Gap}
Stripe Radar, Sift, and Sardine offer mature cloud APIs, but per-transaction pricing and
foreign calibration limit their use for many Nepali PSPs. Auditors also prefer transparent
rules. Flint Secure will fill this gap with \textbf{open, self-hostable} software and weights
adjusted using local fraud reports~\cite{fiu2024}, payment statistics, and a planned pilot data
set (see Section~\ref{sec:validation}).

\section{Proposed Methodology}
\label{sec:methodology}

This section explains \textbf{how} the system will be built: process, tools, main steps, and
design diagrams (Figures~\ref{fig:uml-use-case} and~\ref{fig:uml-seq-score}).

\subsection{Software Development Life Cycle}
We will follow an \textbf{Agile} plan with two-week sprints (Table~\ref{tab:schedule}). Each
sprint will end with testable code and a short supervisor meeting (NCIT Appendix~B log).

\subsection{Tools and Technologies}
The planned stack includes Go with the chi router, two Redis instances, PostgreSQL, NATS
JetStream, Go workers, a JavaScript web SDK, Docker Compose for local testing, k6 for load
tests, and deployment on any VPS or cloud provider available to the team.

\subsection{Repository Structure and Implementation Phases}
Code will be organised in a single repository with folders for the API, workers, web SDK,
and database migrations. Work will proceed in phases:

\subsection{Phase 1: Infrastructure and Middleware}
\textbf{Step 1: Databases and messaging.} Set up PostgreSQL for clients, keys, transactions,
and alerts; Redis for devices and speed-sensitive counters; NATS for events such as new
devices and scored payments.

\textbf{Step 2: Security middleware.} Each API call will pass checks for request ID, client IP,
location lookup (GeoIP), API key, rate limits, and optional signed requests to stop replay
attacks.

\subsection{Phase 2: Device Identification API}
\textbf{Step 3: Web SDK.} The web SDK will collect device signals, hash them on the client,
and call the device identification API.

\textbf{Step 4: Matching on the server.} The server will try, in order: (1)~exact fingerprint
match; (2)~match by saved device ID if the fingerprint changed slightly; (3)~fuzzy match
among similar devices; (4)~create a new device ID if none fit, then notify background workers.

\subsection{Phase 3: Risk Scoring API}
\textbf{Step 5: Load risk data.} Before scoring, the API will read device trust, user habits,
speed counters, recipient risk, and client settings from Redis in one batch.

\textbf{Step 6: Calculate score.} A single scoring function (no database calls inside it)
will combine seven factors (device, amount, time, speed, network, identity, and
recipient) and return a score from 0 to 100 with ALLOW, FLAG, or BLOCK. Hard rules will
force BLOCK for clear SIM swap or datacenter abuse.

\textbf{Step 7: Background updates.} After the response is sent, the system will update speed
counters and publish an event for workers without delaying the customer.

\subsection{Phase 4: Workers and Deployment}
\textbf{Step 8: Background workers.} Go worker services will save transactions, update user
profiles, adjust device trust, and refresh settings in Redis.

\textbf{Step 9: Deployment.} Docker Compose for development; pilot hosting on any VPS or
cloud provider available, with HTTPS and health checks.
